## Deployment

Provisioned by the `infra/aws-fargate` Terraform block (directory-per-
environment under `infra/aws-fargate/envs/<env>/`; state in S3 with native
lockfile locking, no DynamoDB). CI authenticates via **GitHub OIDC** — the
provisioned least-privilege deploy role (`deploy_role_arn` /
`AWS_OIDC_ROLE_ARN`), never long-lived AWS keys.

First-time / infra changes:

```bash
cd infra/aws-fargate/envs/dev
terraform init \
  -backend-config="bucket=<tf-state-bucket>" \
  -backend-config="key=aws-fargate/dev/terraform.tfstate" \
  -backend-config="region=<region>"
terraform plan  -var-file=terraform.tfvars    # review — always
terraform apply -var-file=terraform.tfvars
```

App deploy (what `just deploy <env>` runs — `infra/aws-fargate/scripts/deploy.sh`):

1. **Assume the deploy role via OIDC** (`aws-actions/configure-aws-credentials`
   with `role-to-assume: <deploy_role_arn>`) — no static keys.
2. **Build + push the app image** to ECR (`ecr_repository_url`) tagged with the
   git SHA; the ECR repo is scan-on-push + immutable.
3. **Sync the static site**: `aws s3 sync` the built frontend to the private
   bucket, then `aws cloudfront create-invalidation` on
   `cloudfront_distribution_id`.
4. **`terraform apply`** with `app_image_tag=<sha>` so the task definition
   points at the new image (review the plan for prod).
5. **Roll the service**: `aws ecs update-service --cluster <ecs_cluster_name>
   --service <ecs_service_name> --force-new-deployment`.
6. **Migrations on deploy**: run `alembic upgrade head` (FastAPI) /
   `manage.py migrate` (Django) as a one-off task against the new image
   BEFORE the service finishes rolling — the prod image CMD does not migrate.

## Secrets

| `DATABASE_URL` | infra/aws-fargate (backend) | Generated by Terraform (rds module composes it from the RDS host + generated password with the driver-correct SSL param) and stored in Secrets Manager; injected into the task via `valueFrom`. |
| `JWT_SIGNING_KEY` | infra/aws-fargate (backend auth) | Generated once by Terraform (secrets module) and stored in Secrets Manager; injected via `valueFrom`. Rotating it invalidates live tokens — plan a window. |
| `SMTP_USERNAME`, `SMTP_PASSWORD` | infra/aws-fargate (backend email) | Secret containers created by Terraform; set the real relay values out-of-band (console/CLI), then they inject via `valueFrom`. `SMTP_HOST`/`EMAIL_FROM` are non-secret `app_environment`. |
| `AWS_OIDC_ROLE_ARN` | CI (GitHub Actions) | Terraform output `deploy_role_arn` — the least-privilege role CI assumes via OIDC (`role-to-assume`). No stored AWS keys. |

## Maintenance

- **Verify without applying**: `terraform fmt -check -recursive`, `terraform
  validate`, `terraform plan`, and `checkov --directory infra/aws-fargate
  --framework terraform` (exit 0 by design; documented inline skips only).
  `plan` is offline-safe (`account_id` is a variable, provider `skip_*` flags,
  no plan-time data sources) — run it on every `infra/` PR.
- **Secrets rotation** is manual (no AWS-native rotation Lambda for a signing
  key / relay creds): rotate on suspected exposure or on a calendar per
  `secrets-management.md`. Rotating `DATABASE_URL` means rotating the RDS
  master password and re-storing.
- **Cost watch**: NAT gateways (one per AZ — drop `az_count` to 1 for a cheap
  starter), the ALB, and RDS are the bill-drivers. `deletion_protection` is on
  for RDS — disable it deliberately before a real teardown.
- **Opt-ins** a hardening pass turns on (each an inline `checkov:skip` today):
  WAF on the ALB/CloudFront, access-logging buckets, RDS Multi-AZ
  (`var.multi_az`), S3 cross-region replication, a CloudFront custom domain +
  cert (pins TLS 1.2).
