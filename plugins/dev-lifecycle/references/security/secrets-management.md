# Secrets management

Where secrets live, per environment, and how to get each one. This is the canon behind a scaffolded project's root README **Secrets (and where to get them)** section (`references/authoring/documentation-standard.md`) — every block/recipe's secrets subsection follows this doc's pattern.

## Version check (do this first)
Not a library reference — no version axis. Re-verify the AWS Secrets Manager/SSM and GitHub OIDC mechanics against current docs when this file is revised; the mechanics are stable but service defaults change.

## `.env.example` only, never real `.env`
- Every project commits a `.env.example` with every variable name and a placeholder or description — never a real value. `.env` itself is gitignored from the project's inception (in the scaffolded `.gitignore`, not added later).
- A secret that leaks into a commit is treated as compromised, not just removed — rotate it, don't just delete the line. `git-secrets`/`gitleaks` (or the CI dependency-scanning step in `references/security/secure-baseline.md`) catches this before merge, not after.
- Local dev reads `.env` via the framework's standard loader (`pydantic-settings`, `python-dotenv`, Next.js's built-in `.env.local`) — no bespoke secret-loading code per block.

## Local vs CI vs prod
| Environment | Where secrets live | How they're injected |
| --- | --- | --- |
| Local dev | `.env` (gitignored, developer-local) | Loaded by the framework's settings/env loader at process start. |
| CI (GitHub Actions) | GitHub Actions **encrypted secrets** (repo or environment-scoped) | Injected as job env vars; never echoed to logs. Cloud credentials use OIDC (below), not stored keys, wherever the target supports it. |
| Prod / staging (AWS) | **AWS Secrets Manager** (rotated, structured secrets — DB creds, API keys) or **SSM Parameter Store** (`SecureString`, simpler config/flags) | Injected at container start — ECS task definitions reference the secret ARN directly (`secrets` block), the app never fetches and writes it to disk. |

Secrets Manager vs SSM Parameter Store: reach for Secrets Manager when the secret needs rotation or is a structured credential (DB username+password+host); SSM `SecureString` is fine for a single API key or flag that doesn't need automatic rotation. Both are encrypted with KMS; neither is a substitute for scoping IAM access to who can read them.

## Rotation
- Rotate on a schedule for anything Secrets Manager can rotate automatically (RDS credentials via the built-in rotation Lambda) — enable it at provisioning time, not as a follow-up.
- Rotate immediately, unscheduled, on any suspected exposure (leaked in a log, committed to git, a departing contractor had access) — treat rotation as cheap and routine, not an incident-only action.
- Third-party API keys (Stripe, etc.) without automatic rotation: rotate on a calendar reminder (e.g. every 90 days) and whenever a team member with access leaves.

## GitHub OIDC over long-lived keys
- CI never stores a long-lived AWS access key/secret pair. Instead, configure a GitHub Actions **OIDC identity provider** in the AWS account and an IAM role with a trust policy scoped to this repo (and branch/environment if warranted); the workflow assumes the role via `aws-actions/configure-aws-credentials` with `role-to-assume`, no static keys anywhere.
- Scope the assumed role to least privilege for what that workflow actually does (e.g. `terraform plan` read-only vs an `apply` role with write access) — one broad CI role defeats the purpose.
- This is the same posture `references/infra/aws.md` and `references/infra/terraform.md` require for IaC deploys; OIDC is the mechanism, this doc is where it's documented per-project.

## Per-secret documentation pattern
Every secret a project needs gets one row in the root README's Secrets table, following this pattern — **what it is, and where to obtain it**, never the value:

| Secret | Used by | Where to get it |
| --- | --- | --- |
| `DATABASE_URL` | backend | Local: `.env.example` default (docker-compose Postgres). Staging/prod: generated by Terraform, stored in Secrets Manager. |
| `STRIPE_SECRET_KEY` | payments recipe | Stripe Dashboard → Developers → API keys (test key for dev/CI, live key only in prod's Secrets Manager). |
| `STRIPE_WEBHOOK_SECRET` | payments recipe | Stripe Dashboard → Developers → Webhooks → select endpoint → Signing secret. |
| `AWS_OIDC_ROLE_ARN` | CI (GitHub Actions) | Output of the Terraform `infra/` stack that provisions the CI IAM role. |
| `JWT_SIGNING_KEY` | backend auth | Generated once at provisioning (`openssl rand -hex 32`), stored in Secrets Manager — never regenerate without planning a token-invalidation window. |

Each block/recipe that introduces a secret adds its own row(s) to its doc fragment (`references/authoring/documentation-standard.md`); the root README aggregates them. See `references/security/secure-baseline.md` for the rule this doc is the mechanics behind, and `references/security/payments-security.md` for the Stripe-specific keys above.
